# Debug Session: bot-running-not-working

- Status: OPEN
- Started: 2026-06-04
- Symptom: контейнер и оба бота стартуют без фатальных ошибок, но в пользовательских проверках ожидаемое поведение не наблюдается.

## Initial Evidence

- `docker compose logs -f bot` показывает успешный старт Telegram polling и Discord session.
- Конфигурация читается через Docker environment.
- Явной panic или `invalid token` в логах нет.
- `/start` без payload стабильно отвечает экраном без активных привязок.
- При открытии deep link пользователь видит `Ссылка привязки недействительна или уже устарела.`, что соответствует ветке `GetInviteToken() == nil`.

## Hypotheses

1. Бот запущен, но тестируется сценарий, который не покрывается текущими условиями в коде, поэтому обработчик просто не срабатывает.
2. В Discord у приложения не включены нужные privileged intents, и часть событий не доходит, несмотря на успешный `ready`.
3. В Telegram бот не имеет доступа к целевому чату или пользователь проверяет не тот чат/не тот flow привязки.
4. Данные/статус формируются, но не отправляются из-за runtime-ошибки в конкретной ветке, которую мы пока не видим без дополнительной инструментализации.
5. Пользовательский сценарий "не работает" связан не с запуском сервиса, а с конфликтом ожиданий: например, DM приходит только при отсутствии mapping и при наличии подходящих участников в voice-канале.

## Narrowed Analysis

- Гипотеза о неверных кредах на текущий момент не подтверждается: при неверных токенах бот обычно не проходит этап старта Telegram/Discord.
- Наиболее вероятная причина: используется старый или уже отсутствующий invite token, либо новый token не создается, потому что не выполняются условия отправки DM из Discord.

## Evidence-Based Conclusion

- После полной очистки `data/` были удалены и `invite_tokens`, и `chat_user_mappings`.
- Симптом `Ссылка привязки недействительна или уже устарела.` согласуется с пустой таблицей `invite_tokens`.
- Симптом `Сейчас у тебя нет активных привязок` согласуется с пустой таблицей `chat_user_mappings`.
- В текущей логике Discord DM с deep link строится только для чатов, которые уже представлены привязанными участниками в voice-канале. При пустой базе такой участник отсутствует, поэтому новый DM не генерируется.
- Вероятная корневая причина: после вайпа БД система потеряла bootstrap-состояние и больше не может выдать первую привязку без восстановления данных или ручного seed-а.

## New Runtime Evidence

- В логах есть повторяющаяся ошибка `unable to open database file (14)` на этапе `initialize sqlite schema`.
- Позже контейнер все же стартует успешно, значит креды и сетевое подключение к Telegram/Discord в целом рабочие.
- В предоставленном фрагменте после успешного старта нет логов о новых `VoiceStateUpdate`, отправке DM или ошибках Discord DM.

## Updated Assessment

- Гипотеза о проблеме с кредами: REJECTED.
- Гипотеза о проблеме с доступом к SQLite-файлу/директории: CONFIRMED.
- Гипотеза о недостижении ветки отправки DM при текущем тесте: INCONCLUSIVE, но вероятна, так как в логах нет признаков обработки нового входа пользователя в voice после старта сервиса.

## Direct Host Evidence

- Команда `ls -la data` на хосте возвращает `Permission denied (os error 13)`.
- `sqlite3 data/bot.db ...` на хосте тоже не может открыть файл базы.
- Это подтверждает, что текущий пользователь Debian-ноутбука не имеет достаточных прав на директорию `data` и/или `bot.db`.
- Следующий шаг: восстановить права на `data`, затем повторно проверить содержимое БД и только после этого снова тестировать Discord DM flow.

## Database State After Permission Fix

- `select count(*) from chat_user_mappings;` вернул `0`.
- `select count(*) from invite_tokens;` вернул `0`.
- Следовательно, текущая БД фактически пустая: в ней нет ни одной существующей привязки и ни одного активного invite token.
- При такой БД Discord-flow не сможет построить `eligibleChatInvites`, поэтому DM для первичной привязки не будет отправлен.

## Runtime Evidence From Instrumentation

- `VoiceStateUpdate` действительно приходит: есть лог `[DEBUG] voice state update received` для `discord_id=583958031511977994`.
- Для этого пользователя `mapping_count=0` и `allowed_mapping_count=0`, то есть привязки в БД нет.
- В канале в момент проверки `participant_count=1`, то есть бот видит только самого тестового пользователя.
- После исключения тестового пользователя из расчета eligible invites остается `0` кандидатов, поэтому `invite_count=0`.
- Следовательно, текущая логика не отправляет DM, и это поведение соответствует коду, а не ошибке доставки.

## Minimal Fix Applied

- Изменена только логика bootstrap для `single-chat` режима.
- Если `eligible invites` пусты и в конфигурации задан ровно один целевой Telegram-чат, бот теперь создает invite для этого единственного чата без необходимости видеть уже привязанного участника в voice.
- Поведение для `multi-chat` режима не изменено: там по-прежнему используются только chat-scoped invites, выведенные из уже привязанных участников канала.
- Временная debug-инструментализация оставлена для post-fix проверки.

- После этого добавить минимальную инструментализацию в нужную ветку выполнения и собрать runtime-доказательства.
